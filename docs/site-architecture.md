# Site architecture — JFC Renovation

This document explains how the repository is organized so future human or AI contributors can work without damaging the source-of-truth structure.

## High-level model

JFC Renovation is organized around four layers:

1. **Editable sources** — the files humans and AI should normally edit.
2. **Public assets** — portfolio photos, portrait, styles and client-facing scripts.
3. **Validation scripts** — checks that protect pages, links, assets, layout and repository hygiene.
4. **Documentation and governance** — files that explain what to edit, what not to edit, and how to validate changes.

## Branch model

- `main` is the published branch.
- Important changes should use a short dedicated branch.
- PRs should start as draft when the change is not yet visually or technically validated.
- Do not merge unless checks are green and the user explicitly validates the change.

## Source-of-truth files

### Pages

```txt
index.html
services/index.html
realisations/index.html
a-propos/index.html
contact/index.html
devis/index.html
```

These files are the editable page sources.

### Styles

```txt
style.css
assets/mobile-fixes.css
```

`style.css` contains the global design system. `assets/mobile-fixes.css` contains later visual/mobile fixes and shared component refinements.

### Scripts

```txt
assets/whatsapp-direct.js
scripts/jfc-site-tests.js
scripts/validate-repository-hygiene.js
```

`assets/whatsapp-direct.js` controls mobile WhatsApp app opening with fallback. `scripts/jfc-site-tests.js` is the consolidated browser/static test runner.

### Public assets

```txt
assets/photos/**
assets/about-portrait.svg
```

Portfolio photos must remain curated, readable and linked from active pages. Avoid leaving unused confusing duplicates.

### Build output

```txt
dist/**
```

`dist/` is generated by `npm run build`. It is not the editing surface.

## Validation layer

Important validators and tests include:

```txt
scripts/validate-repository-hygiene.js
scripts/jfc-site-tests.js
```

The permanent CI workflow is:

```txt
.github/workflows/site-tests.yml
```

It runs static validation and browser regression jobs in the same general pattern as the Med Nykuto consolidated workflow.

## Visual testing expectations

The site must be checked at least on:

- mobile Safari-shaped viewport;
- desktop Chromium viewport;
- visual/accessibility smoke pass;
- gallery lightbox behavior;
- home carousel behavior;
- contact links and WhatsApp links;
- broken images and missing alt text;
- horizontal overflow.

## Repository hygiene

The repository must not accumulate:

- temporary debug workflows;
- archives committed to the repository;
- stale copies of old pages or assets;
- backup dumps at runtime paths;
- duplicated confusing photos;
- generated output treated as editable source.

Suspicious historical material should be removed or documented only after explicit validation.

## AI workflow checklist

Before editing:

1. Read `SOURCE_OF_TRUTH.md`.
2. Read `AGENTS.md`.
3. Read `.github/copilot-instructions.md`.
4. Read this file.
5. Identify whether the target file is source, generated output, validator, documentation, workflow, or asset.

Before opening a PR:

1. Confirm the change is narrow.
2. Mention source-of-truth impact.
3. Mention tests expected to run.
4. Keep the PR as draft until CI is green if the change is not yet validated.

Before merging:

1. Confirm CI is green.
2. Confirm the user approved the visual result when the change affects the UI.
3. Do not merge if there are unresolved source-of-truth doubts.
